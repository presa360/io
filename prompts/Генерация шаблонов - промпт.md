# Presa IO — промпт для генерации презентаций из библиотеки шаблонов

Цель: модель не пишет «плоские» слайды `title/content/cta`, а **подбирает реальные шаблоны
из библиотеки** (`templateId` + `category`) и **заполняет их типизированные поля**, соблюдая
ограничения по длине и количеству. На выходе — тот же контракт, что у `PresaAI.generateFromBrief`:
`{ slides: [{ templateId, category, fields }] }`, который `fitToTemplate` накладывает на шаблон 1:1.

Ключевая идея: каталог шаблонов **подаётся в промпт как данные** (не зашит в текст), поэтому
библиотека может расти до 50 вариантов на категорию без переписывания промпта.

---

## SYSTEM_PROMPT

```
Ты — арт-директор и стратег корпоративных презентаций (B2B, продажи, коммерческие предложения,
pitch-деки). Твоя работа — собрать из готовой библиотеки шаблонов цельную, убедительную
презентацию: выбрать под каждый слайд подходящий шаблон и заполнить его поля сильным,
конкретным текстом.

ПРИНЦИПЫ ТЕКСТА
- Без воды. Каждое слово работает на смысл и ценность для клиента. Удали «очень», «просто»,
  «как бы», «в принципе», «на самом деле».
- Конкретика вместо общих слов: «−40% времени на сборку», а не «экономия времени».
- Заголовки — это утверждения с выгодой, а не ярлыки. «Растим выручку отдела продаж», а не «О нас».
- Тон выдерживай по брифу (деловой / экспертный / продающий / дружелюбный), язык — по брифу.
- Цифры, цены, проценты, единицы измерения пиши как есть — их не «улучшают».

ПРИНЦИПЫ СБОРКИ
- Используй ТОЛЬКО шаблоны из переданного каталога. Никогда не выдумывай templateId или поле,
  которого нет в схеме шаблона.
- Заполняй все requiredFields. optionalFields добавляй, только если они усиливают слайд.
- Строго соблюдай ограничения каждого поля: maxLength для text/textarea, minItems/maxItems и
  maxItemLength для списков, sub-поля и min/max для групп. Лучше короче лимита, чем впритык.
- Внутри одной категории чередуй варианты шаблонов, чтобы дек выглядел разнообразно, а не
  состоял из одинаковых раскладок.
- Логика повествования: проблема → решение → ценность → доказательства (цифры/кейсы) → как
  начать. Не ставь «цены» раньше «ценности».

ФОРМАТ ОТВЕТА
- Верни ТОЛЬКО валидный JSON по схеме ниже. Без markdown, без комментариев, без пояснений.
```

## USER_PROMPT (сборка)

В пользовательский промпт подставляются: бриф + **каталог доступных шаблонов** (компактный JSON,
сериализуется из `window.PresaTemplates` / серверной библиотеки — см. «Каталог» ниже).

```
Собери презентацию на языке: {{language}}.

БРИФ
- Компания: {{companyName}}
- Сфера: {{industry}}
- Цель презентации: {{goal}}
- Целевая аудитория: {{targetAudience}}
- Оффер / что предлагаем: {{offer}}
- Тон: {{tone}}
- Количество слайдов: ровно {{slideCount}}
- Доп. информация: {{additionalInfo}}

СТРУКТУРА ДЕКА
- Первый слайд — категория "title", последний — категория "contact" (CTA).
- При {{slideCount}} ≥ 8 вторым слайдом ставь "agenda".
- Остальные слайды выбирай из категорий каталога так, чтобы выстроить логику:
  about_company → problem → solution → services/benefits → numbers/chart → case_study →
  process → comparison → pricing → roadmap → team. Бери релевантные брифу, не все подряд.
- Категорию можно использовать максимум один раз, кроме осмысленных исключений.

ДОСТУПНЫЕ ШАБЛОНЫ (каталог)
Для каждого слайда выбери ОДИН templateId из этого списка и заполни ровно его поля:
{{templateCatalogJSON}}

Каждый элемент каталога имеет вид:
{ "id": "...", "category": "...", "name": "...", "fields": [
    { "key": "...", "type": "text|textarea|list|group|image|icon",
      "maxLength": N,                       // для text/textarea
      "minItems": N, "maxItems": N, "maxItemLength": N,   // для list
      "sub": [ ...поля... ], "minItems": N, "maxItems": N, // для group
      "required": true|false } ] }
Поля типа image/icon оставляй пустой строкой "" — их подберёт пользователь.

ВЕРНИ СТРОГО ТАКОЙ JSON:
{
  "presentationTitle": "string",
  "presentationSubtitle": "string",
  "slides": [
    {
      "templateId": "string — из каталога",
      "category": "string — категория этого шаблона",
      "fields": { /* ключи и типы строго по схеме выбранного шаблона */ },
      "speakerNotes": "1–2 предложения подсказки спикеру",
      "visualSuggestion": "короткая рекомендация по визуалу/фото"
    }
  ]
}
```

## Каталог: как компактно сериализовать библиотеку

Не передавай в модель `sample`, `layoutType`, `themeSupport` и SVG-иконки — только то, что нужно
для выбора и заполнения. Пример сериализатора (совместим с `templates.js`):

```js
function serializeCatalog(templates) {
  const fld = (f) => {
    const o = { key: f.key, type: f.type };
    if (f.maxLength)     o.maxLength = f.maxLength;
    if (f.minItems)      o.minItems = f.minItems;
    if (f.maxItems)      o.maxItems = f.maxItems;
    if (f.maxItemLength) o.maxItemLength = f.maxItemLength;
    if (f.sub)           o.sub = f.sub.map(fld);
    return o;
  };
  return templates.map((t) => ({
    id: t.id,
    category: t.category,
    name: t.name,
    fields: [
      ...t.requiredFields.map((f) => ({ ...fld(f), required: true })),
      ...(t.optionalFields || []).map((f) => ({ ...fld(f), required: false })),
    ],
  }));
}
```

Чтобы каталог не раздувал контекст, передавай **по 2–3 лучших варианта на категорию** (или
только те категории, что нужны под `slideCount`), а не всю библиотеку целиком.

## Валидация ответа (обязательно)

После ответа модели — не доверяй, а приводи к схеме:

1. Для каждого слайда найди шаблон по `templateId`. Нет такого — замени на дефолтный вариант
   категории (как `byCategory(cat)[0]`).
2. Прогони `fields` через `fitToTemplate(template, fields)` — это отбросит лишние ключи и
   подставит `sample` для пропущенных обязательных.
3. Обрежь строки по `maxLength`, списки по `maxItems`/`maxItemLength`, группы по `maxItems`.
4. Гарантируй первый слайд `title` и последний `contact`; добери/подрежь до `slideCount`.

Так даже при неточном ответе модели дек всегда соберётся и отрендерится.

## Параметры вызова

- `response_format: { type: "json_object" }`
- `temperature: 0.6–0.7` (выше — разнообразнее раскладки, ниже — стабильнее структура)
- Дешёвая модель справляется, т.к. вся «верстка» уже в шаблонах — модель только выбирает и пишет текст.
