# Presa — заметки по проекту и журнал изменений

Документ описывает, что это за проект, как он устроен, и какие правки были внесены
в ходе работы (с причинами и проверками).

---

## 1. О проекте

**Presa** — конструктор корпоративных презентаций с ИИ. Пользователь заполняет короткий
бриф или вставляет сырой текст, система разбирает его на слайды, даёт живой предпросмотр
16:9 и экспортирует готовый **PPTX**. Стиль дизайна — red/white tech SaaS (акцент `#E5322B`).

Проект существует в двух воплощениях:

### HTML/React-прототип (корень + `assets/`) — работает без сборки
Открывается прямо в браузере через CDN (React 18 + Babel standalone).

| Файл | Назначение |
|------|-----------|
| `index.html` | Лендинг (`assets/home.jsx`) |
| `generator.html` | **Главное приложение** — конструктор: визард → редактор слайдов → экспорт |
| `probe-all.html` | Витрина всей библиотеки шаблонов |

Ключевые модули в `assets/`:
- `templates.js` — библиотека шаблонов (данные): категории, поля, sample, `RENDER_CAP`, `meta()`.
- `slide-layouts.jsx` + `layouts-construction/pack2/pack3.jsx` — рендер раскладок (по `layoutType`).
- `constructor.jsx` + `cx-*.jsx` — редактор (визард, стейдж, панели, модалки).
- `cx-wizard-steps.jsx` — **Промт-менеджер**: разбор сырого текста на слайды.
- `cx-ui.jsx` — общий UI + `remapFields` (перенос контента при смене варианта шаблона).
- `ai-helpers.js` — детерминированные заглушки ИИ (работают без ключа).
- `pptx-render.js` / `pptx-export.js` — экспорт в PPTX (pptxgenjs).

### Next.js-исходники (`nextjs-source/`) — для реального деплоя
Полноценный MVP: Next.js 14 · TypeScript · Tailwind · pptxgenjs. Настоящие API-роуты
(`generate-presentation`, `export-pptx`), вызов OpenAI-совместимого API. Деплой на Vercel.

### Архитектурные принципы
- **Один контракт полей** питает редактор, превью и PPTX-экспорт.
- **`layoutType`** — единый ключ диспетчеризации рендера в трёх местах.
- Метаданные шаблонов **выводятся**, а не хранятся (`meta()`), поэтому не устаревают.

## 2. Журнал изменений

### 2.1. Синхронизация `RENDER_CAP` с фактической обрезкой
**Файлы:** `assets/templates.js`, `assets/slide-layouts.jsx`
**Причина:** реестр `RENDER_CAP` (сколько элементов раскладка реально рисует) местами
расходился с фактическим `.slice()` и с лимитами полей — скрытые «мины» на будущее.

- `slide-layouts.jsx` — `L_bullets`: `.slice(0, 6)` → `.slice(0, 5)` (совпало с `RENDER_CAP` и полем).
- `templates.js` — `table: 10` → `table: 6` (реестр обещал больше, чем допускает поле).
- `templates.js` — добавлены `cards_band: 6`, `feature_rows: 5` (enforced-раскладки без записи в реестре).

**Проверка:** синтаксис OK; библиотека грузится (161 шаблон); `meta()` считается корректно.
Видимое поведение не изменилось — устранены скрытые рассинхроны.

### 2.2. Качество разбора текста на слайды (Промт-менеджер)
**Файл:** `assets/cx-wizard-steps.jsx`
**Причина:** плохой разбор сырого текста. Найдено эмпирически (прогон реального кода в Node).

Исправленные баги:
1. **Потеря контента при одной секции** — вставка «заголовок + список» давала 1 слайд
   (только обложку), остальное терялось. Теперь единичная секция тоже даёт контент.
2. **Предложение как заголовок + дубль в теле** — сырой абзац клал целое предложение
   в title и дублировал в текст. Добавлены `shortTitle`, `CAT_TITLE` — чистые короткие заголовки.
3. **Обложка = обрезанное предложение** — добавлен `coverFromProse` (разбивает «Бренд — тэглайн»).
4. **Строки-«точки» как границы предложений** — пунктуационные строки-артефакты вставки
   теперь пропускаются.
5. **Игнор переносов строк** — если во вставке уже ≥3 строк, это список; ветка «стена текста»
   больше не режет его по предложениям.
6. **«Метка: описание» → карточки** — ряд из ≥3 таких пунктов рендерится card-грид
   (заголовок+текст: `problem-03`/`solution-04`/`benefits-01`/…), а не плоским списком.

**Проверка:** реальный кейс SCM (был 4 слайда → стал 1 слайд с карточками); регрессы
(markdown-секции, нумерованный список, стена текста, штатный `PROMPT_EXAMPLE`) — без изменений.

### 2.3. Новые варианты шаблонов «Решение»
**Файл:** `assets/templates.js`
**Причина:** расширить визуальное разнообразие категории solution.

Добавлены 5 вариантов (было 10 → стало 15):

| ID | Раскладка | Что это |
|----|-----------|---------|
| `solution-11` | `service_icons` | Карточки с иконками (сетка) |
| `solution-12` | `service_row_icons` | Строки с иконками |
| `solution-13` | `service_blocks` | Крупные блоки |
| `solution-14` | `service_flagship` | Ключевой блок крупно + опоры |
| `solution-15` | `benefits_check_rings` | Чек-лист с кольцевым акцентом |

Контракты полей скопированы 1-в-1 с проверенных шаблонов (`services-04/10/12/09`, `benefits-09`),
все `layoutType` имеют рендерер и запись в `RENDER_CAP`.

**Проверка:** синтаксис OK; 161 шаблон; `meta()` и бейджи корректны для всех пяти.

### 2.4. Перенос контента при смене варианта: «вставляет только в заголовок»
**Файл:** `assets/cx-ui.jsx`
**Причина:** при переключении слайда на вариант с карточками `remapFields` → `coerceItem`
клал весь буллет «Метка: описание» в заголовок карточки, а поле `text` оставлял пустым.

**Фикс:** `coerceItem` разбивает строку «Метка: описание» (или «Метка — описание») на
заголовок + текст — только когда у карточки есть второе текстовое поле и метка короткая (≤7 слов).

**Проверка (реальный `remapFields`):** заголовок и текст заполняются во всех карточных вариантах;
регрессы (буллеты без двоеточия → только heading; обратный перенос card→bullets) — целы; синтаксис OK.

### 2.5. Многосекционный текст с разделителями-«точками»
**Файл:** `assets/cx-wizard-steps.jsx`
**Причина:** большой структурированный текст (≈10 разделов вида «Заголовок раздела» +
пункты «Метка: описание», разделённые строками-«.») давал **3 слайда-каши** — все разделы
сливались в один, потому что заголовки-разделители не распознавались.

Исправления:
1. **Строки-«.» = разрыв абзаца** (`prevBlank = true`) — после них короткая строка проверяется
   как заголовок раздела.
2. **`labeledContent`** — строка «Метка: длинное описание» (>45 симв., ≥4 слов после двоеточия)
   считается контент-буллетом, а не заголовком, даже если начинается с ключевого слова
   («Этап настройки: …» — контент; «Кейс: X» — заголовок раздела).
3. **`contentSegsFrom`** — интро-пункты до первого заголовка и одиночная секция больше не теряются
   в обложке; заголовок обложки передаётся как `hint` для классификации.

**Проверка:** тот же текст (≈10 разделов) → **9–10 слайдов**, по одному на раздел, карточные сетки
и процессные шаги с сохранением всех пунктов. Регрессы (markdown, нумерованный процесс,
стена текста, `PROMPT_EXAMPLE`) — целы.

### 2.6. Разнообразие карточных раскладок + новые варианты «Решение»
**Файлы:** `assets/templates.js`, `assets/cx-wizard-steps.jsx`
**Причина:** многосекционный «Метка: описание» контент рендерился одним и тем же `solution-04`
на каждом разделе — монотонно.

- Добавлены 2 варианта: `solution-16` (`service_list_panel`) и `solution-17` (`service_minimal`).
  Всего вариантов solution: 10 → **17**; шаблонов в библиотеке: 161 → **163**.
- **Ротация раскладок** — `labeledCardsPlan` и faithful-ветка карточек выбирают шаблон из пула
  (`VARIETY['cards:<cat>']`) через `vpick` (наименее использованный `layoutType` по всему деку).
  Пул solution: `cards → service_icons → service_blocks → service_list_panel → service_minimal → badges → check_rings`.
- **`cardIcon`** — карточкам присваивается иконка по смыслу (по ключевым словам: поставки→box,
  риск→shield, аналитика→chart и т.д.), иначе icon-раскладки рисовали одну и ту же иконку.

**Проверка:** SCM-дек — каждый раздел теперь на своей раскладке с осмысленными иконками;
`solution-16/17` грузятся и рендерятся; регрессы целы (ротация срабатывает только на
«Метка: описание» контенте ≥3 пунктов, обычные буллеты не затронуты).

### 2.7. Пулы ротации для about_company, process, numbers
**Файл:** `assets/cx-wizard-steps.jsx`
**Причина:** повторяющиеся разделы этих категорий рендерились одинаково.

- Пулы: `cards:about_company` (`about-company-03/10/13`), `process:steps`
  (`process-02/04/05/06`, перенос полей через `V()`/`remapFields`), для numbers —
  существующий `numbers:s` (`numbers-01/04/06`).
- Подключение: `about_company` → в `CARDS_POOL_CATS`; `labeledCardsPlan` возвращает
  `null` для `process` (процесс → шаги); faithful-ветки numbers/process крутят через `V()`.
- **Сопутствующие фиксы разбора** (без них ротация теряла контент):
  - `contentSegsFrom` — первый раздел с заголовком и ≥2 строк тела больше не теряется
    в обложке (тело → контент, заголовок → `hint`); подзаголовок обложки в этом случае пуст.
  - `classifyBody`/`isProse` — строка «Метка: описание» никогда не считается вводной прозой
    (иначе длинные пункты уходили в `desc`, короткие — в буллеты, часть шагов терялась).
  - `labeledContent` — порог переведён на «≥4 слов после двоеточия» (короткие пункты вроде
    «Передача: Обучаем и передаём…» больше не становятся ложными заголовками; «Кейс: X» — остаётся).

**Проверка:** multi-текст (3 process + 2 about + numbers) → раскладки чередуются
(`process-02→process-04`, `about-company-10→13`), все шаги/карточки/метрики сохранены,
«stray»-слайдов нет; регрессы (SCM-дек, нумерованный процесс, `PROMPT_EXAMPLE`,
plain/markdown, обложка «Компания Ромашка» с подзаголовком) — целы.

### 2.8. Экспорт PPTX не открывался («файл повреждён»)
**Файл:** `assets/pptx-render.js`
**Причина:** иконки в карточках рендерились как **SVG-картинки** (`iconUri` → `data:image/svg+xml`
→ `slide.addImage`). pptxgenjs растеризует SVG через браузерный `<canvas>` (`new Image()` +
`canvas.toDataURL`) — хрупкий путь: при tainted-canvas или нулевом размере SVG промис
записи **отклоняется**, и скачивается битый/неполный файл. После ротации (2.6/2.7) деки стали
полны icon-раскладок, поэтому экспорт ломался почти всегда.

**Фикс:** добавлен `iconMark()` — иконка рисуется **нативной фигурой** PPTX (скруглённый квадрат
в цвете иконки). Все 4 вызова `addImage(iconUri(...))` заменены на `iconMark(...)`. SVG-картинки
из экспорта исчезли полностью → зависимости от `canvas` больше нет.

**Проверка (реальный pptxgenjs 3.12 в Node):** до фикса 15 icon-шаблонов роняли экспорт
(«Image is not defined»); после — **все 163 шаблона** собираются без ошибок, XML/связи/
Content_Types валидны. Реальный SCM-дек: было 16 media (8 SVG), стало **0 media**, файл
структурно валиден и не требует canvas.

> Побочно замечено (не чинилось — не ломает файл): стоковые фото передаются в `addImage`
> как `data:` с URL Unsplash → pptxgenjs их молча отбрасывает (картинки просто не попадают
> в слайд). Для реальных фото нужно передавать URL через `path:`.

### 2.9. Карточки «в разброс» — ротация без учёта ёмкости
**Файл:** `assets/cx-wizard-steps.jsx`
**Причина:** ротация card-раскладок (`vpick`) выбирала раскладку по разнообразию,
не глядя на число карточек. Раздел на 5–6 пунктов мог попасть в раскладку на 4
(`solution_badges` cap 4, `about_geo_icons`/`about_lead_cards` cap 4, `benefits_check_rings` cap 5)
— карточки вылезали за границы / рассыпались.

**Фикс:** добавлен `vpickCap(key, count, fallback)` — выбирает только среди раскладок пула,
чья `RENDER_CAP >= count` (иначе весь пул как fallback). Применён в `labeledCardsPlan`
и в faithful-ветке карточек.

**Проверка:** SCM-дек — раздел на 5 карточек стабильно уходит на раскладку ёмкостью 6
(`service_blocks`), переполнений нет. DOM-гриды карточек и так count-aware (3/4/5/6
обрабатываются спец-спанами — «сирот» в последнем ряду нет). Экспорт КП-Lenovo: все
9 слайдов — элементы в границах слайда (проверка позиций в EMU).

### 2.10. Финальный CTA-слайд классифицировался как «команда»
**Файл:** `assets/cx-wizard-steps.jsx`
**Причина:** заголовок-призыв «Как начать» не имел contact-сигнала в `HEAD_KW`, а в теле
(«…расчёт под вашу **команду**…») слово «команду» матчило `team` в `SECTION_KW` раньше
`contact` → последний слайд КП рендерился обычным текстом вместо призыва.

**Фикс:** в `HEAD_KW.contact` добавлены `как начать|с чего начать|начать|начнём` — заголовок-CTA
теперь распознаётся до анализа тела. `contact` идёт последним в `HEAD_KW`, а `process`
(«внедрен») — раньше, поэтому «Начать внедрение» остаётся process (регресс проверен).

**Проверка:** КП-Lenovo — слайд «Как начать» → `contact-01` [cta]; «Наш процесс работы» →
`process-02`; дек пересобран в валидный PPTX (9 слайдов, открывается).

## 3. Как проверять изменения

- **Синтаксис:** `node --check assets/templates.js` (и копия `.jsx` как `.js` для JSX-файлов).
- **Логика разбора/переноса:** файлы `cx-wizard-steps.jsx`, `cx-ui.jsx` содержат чистые функции —
  тестируются исполнением в Node со стабами `window`/`React` (см. историю работы).
- **Визуально:** открыть `probe-all.html` (вся библиотека) или `generator.html` (конструктор)
  в браузере. Живой прогон через расширение Claude-in-Chrome — при подключённом расширении.

> Проект не под git — при необходимости зафиксировать историю можно инициализировать репозиторий.
